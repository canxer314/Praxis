# Why AgentOS V9?

> 从第一性原理出发，分析 V8 在复杂连续任务中 token 爆炸的根本原因，以及为什么需要上下文压力感知注入。

---

## 一、不可否认的工程基础

```
事实 1: 1M tokens 是共享资源。AgentOS 注入、系统提示、对话历史、
        用户任务数据、LLM 输出缓冲共享同一个上下文窗口。

事实 2: AgentOS 无法控制其他消费者的 token 消耗。
        - 用户上传的文档: 0-500K tokens (AgentOS 不可控)
        - 对话历史增长: 每轮 +2K-10K tokens (AgentOS 不可控)
        - LLM 输出缓冲: 预留 50K-100K (AgentOS 不可控)

事实 3: 在复杂连续任务中，非 AgentOS 消耗可达 600K-900K tokens。
        → AgentOS 的固定注入可能成为溢出触发器。

事实 4: V8 的全量注入在 90% 的场景中工作良好（上下文使用率 < 50%）。
        但在 10% 的极端场景中（大文档 + 长对话），它会导致崩溃。

事实 5: 一个在 90% 场景中良好、在 10% 场景中崩溃的系统，
        在实际使用中的感知就是"不稳定"。
```

---

## 二、Token 爆炸的根本原因分析

### 2.1 为什么 V8 假设会失效

```
V8 的假设: "AgentOS 注入占 1M 上下文的 3-8%，永远不是瓶颈"
           → 这个假设在"空上下文"场景下是对的

现实场景: 上下文不是从 0 开始的。用户已经进行了 50 轮对话，
          上传了 3 个需求文档，LLM 输出了大量代码和解释。
          
          这时 AgentOS 的 30K 注入就是压死骆驼的最后一根稻草。
          或更糟——AgentOS 注入在系统提示中，最早进入上下文，
          在自动截断时反而最安全。但如果注入在对话历史区域
          （某些 Hook 实现），就可能被截断丢失。

关键误解: "1M 是给 AgentOS 的" → 错。1M 是给整个会话的。
          AgentOS 只是其中一个消费者，而且不是优先级最高的消费者。
```

### 2.2 定量分析：什么时候会爆炸

```
场景: 开发者使用 AgentOS 辅助一个大型软件项目

会话中的数据构成:
  ┌────────────────────────────────────────────────────┐
  │ 系统提示 + 工具定义:                    20K tokens │
  │ AgentOS 注入 (V8 全量, 200 结构):       80K tokens │
  │ 对话历史 (3 小时, 80 轮):             350K tokens │
  │ 用户上传的需求文档 (PDF x3):           280K tokens │
  │ 用户上传的代码库片段:                   120K tokens │
  │ LLM 输出缓冲预留:                      100K tokens │
  │────────────────────────────────────────────────────│
  │ 总计:                                  950K tokens │
  │ 剩余:                                   50K tokens │
  └────────────────────────────────────────────────────┘
  
  50K 剩余 → 只能再支持 ~5 轮对话 → 然后就溢出了。
  
  此时如果 AgentOS 能在压力升高时自动将注入从 80K 压缩到 5K，
  剩余空间就从 50K 增加到 125K → 多支持 ~12 轮对话。
```

### 2.3 Token 爆炸 vs 注意力稀释：两个不同的问题

```
注意力稀释 (V8 尝试解决):
  "1M 上下文中，LLM 注意不到第 500K token 处的关键信息"
  → 解决方案: 排序 (Tier A/B/C)、置信度校准
  → 这是质量问题

Token 爆炸 (V9 解决):
  "1M 上下文快用完了，再加任何东西都会溢出"
  → 解决方案: 压缩 (四级降级)、按需检索
  → 这是容量问题

两者的关系: 
  - 注意力稀释在上下文不太满时也会发生
  - Token 爆炸发生时，注意力稀释一定已非常严重
  - 解决 Token 爆炸的压力感知机制同时缓解了注意力稀释
    (压缩后的注入更精简 → LLM 更容易注意到关键内容)
```

## 三、为什么不是"回到 V7 的选择性注入"

这是 V9 设计中最容易被误解的点。

```
V7 选择性注入:
  决策: "当前场景匹配度 > 0.7 → 注入；< 0.7 → 不注入"
  标准: 场景匹配度
  后果: 匹配错误 → 关键结构完全不可见 → LLM 不知道它的存在

V9 压缩注入:
  决策: "上下文压力 → 压缩详细程度, 不排除任何结构"
  标准: 上下文使用率
  后果: 即使在 Critical 级别:
        - LLM 仍知道所有结构的存在 (索引)
        - LLM 可主动拉取任何结构的完整详情 (recall_structure)
        - 没有信息被永久排除，只是延迟获取

本质区别:
  V7: "你不需要的我不给你"        → 可能漏掉需要的
  V9: "空间不够时我只给你目录"    → 漏不掉，只是需要你主动要
```

## 四、V9 七个改进的逐一论证

### 4.1 上下文压力感知注入

**为什么必须做**：V8 在 10% 的极端场景下会崩溃。不做压力感知，AgentOS 在复杂连续任务中就是不可靠的。

**工程可行性**：
- 上下文 token 估算：已有成熟的 tokenizer，估算精度 > 90%
- 四级压缩：纯文本操作，不需要 LLM 调用
- 压力等级切换：基于简单的阈值判断，零延迟

**不做的后果**：溢出时 AgentOS 注入被截断 → 认知结构丢失 → LLM 退化为无 AgentOS 状态 → 该会话的学习全部浪费。

### 4.2 按需结构检索 (Lazy Loading)

**为什么必须做**：Critical 压力下，1K tokens 只能装索引。LLM 必须有路径获取完整详情。

**工程可行性**：
- recall_structure 作为 tool 注册到 OpenClaw → LLM 可主动调用
- 结构已在 session_start 时全量加载到内存 → 检索延迟 < 10ms
- 语义匹配: 关键字 + embedding 双重匹配

**关键设计决策**：LLM 是否会主动调用 recall_structure？这取决于系统提示中的指令。"当你需要某个认知结构的完整详情时，调用 recall_structure"——如果 LLM 不调用，就退化到纯索引模式。

⚠️ 此处存在不确定性：LLM 主动调用 recall_structure 的频率和质量未经验证。可能需要几轮 prompt 迭代。

### 4.3 注意力遥测

**为什么必须做**：V8 不知道 LLM 实际用了哪些结构。置信度更新可能基于碰巧做对的结果（噪声信号）。

**工程可行性**：
- [STRUCTURE_USED: proto_id] 标记 → 在系统提示中给 LLM 指令
- 解析标记 → 统计每个结构的会话级采用率
- 与置信度对比 → 发现"高置信低采用"的异常结构

**价值**：不只是"知道 LLM 用了什么"——更重要的是发现"哪些结构虽然置信度高但 LLM 从来不用"——这些是需要重点审查的候选退化对象。

### 4.4 验证覆盖扩展

**为什么必须做**：V8 的统计验证器只覆盖 ProtoSequence。75% 的结构类型仍在验证真空中。

**工程可行性**：

```
ProtoRole 验证:
  方法: 角色行为 vs 实际工具调用者模式
  信号: 如果 ProtoRole"医生"定义了行为"问诊+开药+检查",
        但实际工具调用中这些行为分散在 3 个不同角色上
        → 角色定义可能错误
  
  实现: 在 after_tool_call 中记录 tool 的调用上下文（谁发起的）
        与 ProtoRole 的行为列表比对
        → 独立于 LLM 自报告的统计信号

ProtoConcept 验证:
  方法: 对抗性 prompt
  在 session_end 中: "以下是一个已形成的概念。请尝试找出反例:
        这个概念中的哪些特征可能不成立？哪些边界案例被遗漏？"
  如果 LLM 找不到显著反例 → 概念可能正确
  如果 LLM 找到多个反例 → 概念需要修正
  
  这不是完全独立的验证（仍用 LLM），但对抗性角度提供了
  不同于"归纳"的视角。

ProtoPurpose 验证:
  方法: 用户目标完成度关联
  如果 ProtoPurpose 是"患者完成缴费离院"，
  但会话中用户的目标（显式或隐式）与"离院"无关，
  而是"住院治疗"→ ProtoPurpose 可能错误
```

### 4.5 工具-业务步骤映射增强

**为什么必须做**：V8 的 fuzzyMatchScore 基于字符重叠率，在工具名和业务术语之间存在语义鸿沟时完全失效。

**工程可行性**：
- 方案: ProtoStructure 构造时，LLM 为每个步骤输出 `possible_tool_patterns`
- 统计验证器使用 `possible_tool_patterns` 而非步骤 label 做匹配
- 备选: 小型 embedding 模型 (非任务 LLM) → 独立语义匹配

### 4.6 自适应配置

**为什么必须做**：46 个参数无人能调 → 系统无法根据实际使用模式优化。

**工程可行性**：
- 核心参数（置信度阈值、退化阈值、观察次数阈值）根据历史数据自动校准
- 校准方法: 如果运维者经常驳回固化提案 → 自动提高 crystallization threshold
- 安全机制: 自适应范围有限（如 ±20%），运维者可锁定任何参数

### 4.7 主动一致性引擎

**为什么必须做**：V8 只被动等信号。矛盾可能长期未被发现。

**工程可行性**：
- session_end 中调用 LLM 做一次性一致性扫描
- 输入: 同一场景的所有 ProtoStructure
- 任务: "以下是对同一场景的多个认知结构。它们之间是否存在逻辑矛盾？"
- 输出: 矛盾列表 + 受影响的结构 + 建议

## 五、V9 能力可行性矩阵

| 能力 | V8 评估 | V9 评估 | V9 变化 |
|------|--------|--------|--------|
| 上下文压力感知 | ❌ 不存在 | 🟢 **可行** | token 估算 + 阈值判断 (本地逻辑) |
| 四级压缩注入 | ❌ 不存在 | 🟢 **可行** | 纯文本操作，不需 LLM |
| 按需结构检索 | ❌ 不存在 | 🟡 **需验证** | LLM 主动调用频率未知 |
| 注意力遥测 | ❌ 不存在 | 🟢 **可行** | 与预测标记协议同模式 |
| ProtoRole 验证 | ❌ 不存在 | 🟡 **需验证** | 工具调用者信息的可用性取决于 OpenClaw |
| ProtoConcept 验证 | ❌ 不存在 | 🟢 **可行** | 对抗性 prompt (同架构审计模式) |
| 工具映射增强 | V8 有(粗糙) | 🟢 **更准确** | LLM 预标注 + embedding 匹配 |
| 自适应配置 | ❌ 不存在 | 🟢 **可行** | 基于历史数据的统计校准 |
| 主动一致性 | ❌ 不存在 | 🟢 **可行** | session_end LLM 扫描 |

## 六、核心风险

| 风险 | 缓解 |
|------|------|
| 上下文 token 估算不准确 | 使用模型原生 tokenizer + 10% 安全裕度 |
| LLM 不主动调用 recall_structure | prompt 迭代; 降级到定期自动注入关键结构 |
| 压缩注入丢失关键细节 | Tier A 始终保留完整详情; 仅压缩 Tier B/C |
| 自适应配置"调坏"参数 | 限制调整范围(±20%); 保留手动锁定 |
| 验证覆盖扩展引入新的误报 | ProtoRole/Concept 验证的置信度权重初始设为 0.2 (低权重) |

## 七、反向论证

**反方论点**："V9 的七个改进中，只有压力感知和 Lazy Loading 是解决 token 爆炸直接需要的。其余五个改进（注意力遥测、验证扩展、工具映射、自适应配置、一致性检查）可以等 V8 运行并收集真实数据后再决定。一次性做七个改进增加了 V9 的复杂度，延长了实现周期，且五个改进的效果未经验证。"

**回应**：
1. 注意力遥测是压力感知的前提——如果不知道 LLM 实际用了哪些结构，压缩时就不知道该保留什么、该压缩什么。没有遥测，压缩是盲目的。
2. 验证覆盖扩展是 V8 已知但未解决的 gap——V8 的统计验证器覆盖 25% 结构类型，V9 不解决的话，ProtoRole/Concept/Purpose 的置信度信号质量无法改善。
3. 工具映射增强是统计验证器准确率的基础——当前 fuzzyMatchScore 的局限性会随着 ProtoSequence 数量增长而加剧。
4. 自适应配置和一致性检查相对独立，可以推迟到 V9 Phase 3/4。因此 V9 的 4 个 Phase 中，前两个聚焦核心（压力感知 + 遥测 + 验证扩展），后两个做增强（自适应配置 + 一致性检查）。

---

## 八、可证伪预测

1. 如果 100 次会话中上下文使用率从未超过 60% → 压力感知机制未触发 → 其对 V8 的增量价值为零
2. 如果 LLM 主动调用 recall_structure 的频率 < 1 次/会话 → Lazy Loading 策略需要重新设计
3. 如果 [STRUCTURE_USED] 标记的准确率 < 50%（人工验证）→ 注意力遥测不可靠，需要替代方案
