# Why Praxis V5?

> V4 让 Praxis 能推动已知流程。但它不知道"这个流程本身就不对"——它会把任何场景强行塞进最接近的模板。V5 解决：当框架本身是问题的时候，谁来发现、谁来修复。

---

## 一、从 V1 到 V4 的进化链，以及它停在了哪里

```
V1: LLM 无状态 → 需要跨会话持久学习
V2: 学习闭环需要载体 → OpenClaw Memory Plugin
V3: 学习不能只是工具熟练度 → 四维模型 + Curiosity Engine
V4: 学到的知识要驱动行动 → Process Engine + Momentum Engine
V5: ? → ?
```

V4 的终点有一个隐含的墙。Praxis 在这个墙面前会撞上去——不是因为它不够聪明，是因为它**不知道墙的存在**。

---

## 二、墙是什么？5 Whys 分析

### 场景：用户说"帮我跟供应商谈判"

**Why 1**：为什么 V4 处理不好谈判？
→ 因为 V4 的 Process Engine 会把谈判强行匹配到"合同审核流程"（最接近的模板，适配度 0.2）。谈判不是线性步骤——它是利益交换。套用错误框架导致失败。

**Why 2**：为什么 V4 不去构造一个新的认知框架来应对谈判？
→ 因为 V4 的所有子系统都是处理**已分类的已知场景**。ProcessTemplate 可以优化参数（reasonable_wait 从 5 天缩短为 3 天），但不能从"合同审核"变成"谈判协商"——这两个是**结构上不同**的思维模型。

**Why 3**：为什么 V4 不能区分"参数需要调整"和"结构本身不对"？
→ 因为 V4 的检测机制都在**认知结构内部**运作——Action Verification 验证"这一步做对了吗"，Momentum Engine 处理"这一步卡住了怎么办"。没有一个机制站在**结构外部**问："这个结构本身是不是用错了地方？"

**Why 4**：为什么没有结构外部的观察机制？
→ [假设] 因为 V1-V4 的设计方法假设"人类会为每个场景提供正确的认知框架"。ProcessTemplate 是人类预置的，Role 是人类定义的，知识分类是人类设计的。Praxis 在这些框架内优化，但从不敢质疑框架本身。

**Why 5**：追到最底层——V1-V4 的工程哲学和真正的智能之间差了什么？
→ 差了一个**观察自身的视角**。V1-V4 的 Praxis 像一个很能干的员工——你告诉它做什么、怎么做，它做得越来越好。但它永远不会走到你面前说："老板，我觉得我们处理这类事情的方式本身就有问题。我们试试一种完全不同的思路？"

## 三、StructuralGap vs KnowledgeGap

这是 V5 最核心的概念区分：

```yaml
# V3 的知识缺口: "我不懂这个概念"
KnowledgeGap:
  topic: "膜力云数据源架构"
  question: "城乡数据平台是实时API吗？"
  fill_by: "问用户 / 查资料"
  # 填补后: 知识库多了一个事实
  # 解决: "我不知道 X"

# V5 的结构缺口: "我现有的思考框架不足以处理这类场景"
StructuralGap:
  pattern: "反复在谈判类场景中失败，但失败原因不是知识缺失"
  evidence:
    - "5 次谈判场景: 每次用 Process Engine 套，适配度 < 0.3"
    - "ActionVerification 显示 step_decision_accuracy 在谈判场景中 < 0.3"
    - "用户 3 次说'你不理解谈判的微妙之处'"
    - "Momentum Engine 的 nudge/escalate 二元决策不适配谈判中的让步策略"
  hypothesis: "需要一种非线性的、利益交换导向的认知结构"
  fill_by: "构造新认知结构 → 实验 → 验证 → 固化"
  # 填补后: Praxis 多了一种思考方式
  # 解决: "我需要一种新的思维框架"
```

**KnowledgeGap 是关于"信息"的。StructuralGap 是关于"信息组织方式"的。**

---

## 四、为什么 Meta Layer 是必要的

V4 有 6 层 + 4 个子系统。每一层处理一类问题。但没有任何一层负责**"当所有层都不够用时"**的问题。

Meta Layer 就是这一层。它的职责是：

1. **持续监控** 所有 V4 子系统的**适配质量**——不只是单个步骤的成功/失败，而是整个认知框架是否适配当前场景
2. **聚类失败模式** 到结构性原因——"这些失败不是因为缺知识（V3 能处理），不是因为流程卡住（V4 能处理），是因为我的思维框架本身错了"
3. **生成候选的新认知结构**——在人类治理下，提出新的思维框架假设
4. **管理结构实验**——在限定范围内试点新结构，收集效果数据
5. **维护结构注册表**——哪些结构已验证、哪些在实验中、哪些被废弃

---

## 五、V4 → V5 的核心变化

| 维度 | V4 | V5 |
|------|----|----|
| **失败的归因** | 参数问题（proficiency 不够 / 等待时间太长） | 参数问题 **或** 结构问题（框架本身不对） |
| **解决方案** | 调整参数（+0.03 proficiency / -1 天 wait） | 参数调整 **或** 构造新认知结构 |
| **谁发现问题** | Action Verification（在结构内） | **Structural Inadequacy Detector**（在结构外） |
| **谁提出方案** | 人类（设计新模板） | **Cognitive Structure Constructor**（候选）+ 人类（审核） |
| **人类角色** | 设计者（设计新子系统/模板） | **治理者**（审核提案、设定边界、紧急回滚） |
| **认知结构** | 固定（6 层 + 4 子系统） | **可演化**（Cognitive Structure Registry 动态增长） |

## 六、反向论证

**反方论点**："V5 没必要——人类遇到新场景时自己设计新模板就行。让 Praxis 参与自身架构设计，风险太大，收益不确定。"

**为什么这个论点不够**：

1. **Scalability** — V1→V4 的 4 个版本催生了 4 轮人类设计。谈判、竞品分析、风险管理、团队建设、心理咨询...每类新场景都需要人类设计新模板吗？人类是瓶颈。
2. **时机** — 人类通常事后才发现"Praxis 处理不好某类场景"。Praxis 在**执行过程中就能发现**——它实时感受到"我正在强行套用一个不适配的框架"。
3. **粒度** — 人类设计的是大粒度模板。Praxis 在微观交互中学习到的细微结构差异（"催 PM 张三用 casual 语气有效，催架构师李四用正式语气有效"），如果 Praxis 能自己泛化为结构知识（"技术角色偏好正式异步沟通"），比人类手动更新每个角色画像更高效。
4. **但风险确实存在** — 这就是为什么 V5 的三条铁律要求人类治理：任何新结构必须人类批准、必须在限定范围试点、必须可回滚。

**什么条件会改变结论？**
- 如果 Praxis 的使用范围仅限于 3-5 类已知场景 → V4 足够
- 如果人类愿意持续为每个新场景设计模板 → V4 + 人类设计足够
- 如果底层 LLM 的结构创新质量太低（提出的候选结构总是不可用）→ V5 需要重新评估可行性

⚠️ 此处存在不确定性：LLM 能否从"我在谈判场景中重复失败"中归纳出"我需要利益交换模型而不是线性审批模型"？这需要在实现阶段通过实验验证。
