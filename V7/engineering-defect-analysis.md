## 二、三大核心问题

### 1. 验证真空
链路中不存在独立于 LLM 的质量验证信号。LLM 标记 → LLM 归纳 → LLM 审计，全链路自引用。
- **可解性**：🟡 可大幅缓解。引入统计验证（工具调用序列 vs ProtoSequence 预测）作为独立信号源。
- **大上下文影响**：🟡 间接帮助（更多数据使统计验证更准），不能根本解决。

### 2. 接地脆性
Regex + 关键词匹配无法可靠捕获语义显著性。
- **可解性**：🟢 可根本解决。大上下文下可将 SalientElement 提取完全移至 session_end LLM 调用，以完整 transcript 为输入，消除有损压缩。
- **大上下文影响**：🟢 直接解决。可删除 regex 预标记和 PMI 预筛选。

### 3. 人类依赖
学习闭环在多个节点需要人类主动参与，无降级路径。
- **可解性**：🟢 可大幅缓解。置信度门控自动固化 + 隐式批准 + 主动追问 + 异步审批。
- **大上下文影响**：🟡 间接帮助（更准确使自动决策更安全）。

## 三、大上下文窗口的系统性影响

### 可以删除/简化的模块
- `salience-marker.ts` → **删除**（regex 预标记不必要）
- `pattern-detector.ts` → **删除**（PMI 预筛选不必要）
- `scene-matcher.ts` 选择性匹配 → **简化为场景识别**
- `context-builder.ts` 注入策略 → **简化为序列化器 + 重要性排序**
- Token 预算常量 → **改为注意力预算**

### 什么没有被改变
- LLM 无状态性（仍需 Hook 回调做记忆注入）
- LLM 自我验证的结构性盲区（仍需独立信号源）
- 人类注意力稀缺性（仍需自动化降级）
- Hook 同步约束（message_received 仍不能阻塞）
- AgentMemory 单点依赖（仍需本地缓存）

### 新引入的风险
- **注意力稀释**：大上下文 ≠ 大注意力。信息过载可能导致 LLM 忽略关键结构。（基于 "Lost in the Middle" 等注意力衰减研究）

## 四、反向论证

**反方论点**：大上下文只把问题从"信息不足"转移到"信息过多"。验证真空是逻辑问题（同一实体不能自验证），注意力稀释是新风险，静默错误比显式停滞更危险。

**回应**：
1. 统计信号可打破验证真空（真正的独立验证）
2. 注意力稀释可通过关键信息前置 + 层级化组织缓解
3. 静默 vs 显式是风险偏好问题，可通过 GovernancePolicy 配置

**结论置信度**：中等。大上下文大幅改善接地脆性，可简化架构。但未解决验证真空，且引入注意力稀释新风险。

---

## 五、可证伪条件

- regex 预标记在真实对话中准确率 > 60% → 缺陷 2 不成立
- LLM 归纳 ProtoSequence 在 3 次观察后准确率 > 50% → 缺陷 4 不成立
- 用户对审批命令响应率 > 70% → 缺陷 6 不成立
- LLM 注意力衰减在 1M 上下文中对 Praxis 结构注入场景不显著 → 注意力稀释风险消除
