# How does Praxis V6 work?

## 总览：V5 全部保留 + Proto-Cognitive Engine + 层自修改

V6 保留 V5 的全部架构（Meta Layer + V4 六层 + 四子系统），新增 Proto-Cognitive Engine 和层自修改机制。

```
┌──────────────────────────────────────────────────────────────┐
│                   OpenClaw Agent Loop                         │
│                                                               │
│  ┌────────────────────────────────────────────────────────┐  │
│  │              Praxis Memory Plugin (V6)                  │  │
│  │                                                          │  │
│  │  Meta Layer:                                             │  │
│  │  ├─ V5: Structural Detector + Constructor + Registry    │  │
│  │  └─ V6: Proto-Cognitive Engine                          │  │
│  │                                                          │  │
│  │  V4-V5: 六层 + 子系统 (可被 V6 修改)                    │  │
│  └──────────────────────────────────────────────────────────┘  │
└──────────────────────────────────────────────────────────────┘
```

## Proto-Cognitive Engine 四个阶段

### Phase 1: 开放感知（Open Perception）

**目标**：在没有任何分类框架的情况下，标记值得注意的元素。

**核心原则**：**不强行理解。不强行分类。保持认知开放。**

```
┌─────────────────────────────────────────────────────────┐
│              Open Perception Phase                        │
│                                                           │
│  触发: 场景匹配 → 所有 CognitiveStructure 适配度 = 0     │
│                                                           │
│  Step 1: 声明零先验状态                                   │
│  ├─ Praxis 向用户声明:                                   │
│  │  "这是我完全不了解的场景。我会先观察和学习。"          │
│  └─ 设置认知模式: "open_perception" (非"matching")       │
│                                                           │
│  Step 2: 收集原始信息                                     │
│  ├─ 来源:                                                 │
│  │  • 用户消息（文字/语音转写）                            │
│  │  • 位置/环境上下文（GPS, 时间, 周边信息）              │
│  │  • 外部搜索（快速获取场景相关的通用知识）              │
│  │  • 后续互动（用户/协作者的每一条消息）                 │
│  └─ 策略: 广度优先（先收集多种信息），不追求精确          │
│                                                           │
│  Step 3: 标记 SalientElement                              │
│  ├─ 判断"这个元素值得追踪"的依据:                          │
│  │  • 重复出现（同一个词/行为被多次提及）                 │
│  │  • 被用户强调（"注意"、"关键是"、"你必须知道"）       │
│  │  • 出现在序列的转折点（"先X...然后Y"）                │
│  │  • 与其他已知概念有可类比性（"这个像..."）             │
│  │  • 引发用户或协作者的情绪反应                          │
│  └─ 标记但不分类: 只记录"我注意到 X"                     │
│                                                           │
│  Step 4: 生成感知性问题                                   │
│  ├─ 不试图自己找答案——向用户提问                           │
│  │  • "X 是什么？"                                       │
│  │  • "A 和 B 有什么区别？"                              │
│  │  • "这个经常发生还是偶尔？"                            │
│  └─ 问题遵循 Curiosity Engine 的治理（频率/时机）        │
│                                                           │
│  输出: SalientElement[] + 初步感知地图                    │
└─────────────────────────────────────────────────────────┘
```

### Phase 2: 原型构造（Proto-Structure Construction）

**目标**：从 salient 元素的共现模式中，形成带置信度的模糊原型。

```
┌─────────────────────────────────────────────────────────┐
│           Proto-Structure Construction                    │
│                                                           │
│  输入: SalientElement[] + 多次场景观察                    │
│                                                           │
│  Step 1: 检测共现模式                                     │
│  ├─ A 是否总在 B 之前/之后出现？                          │
│  │   → ProtoSequence: [A → B] (置信度 = 共现次数/总次数) │
│  │                                                        │
│  ├─ 几组元素是否总被同一个人/角色处理？                   │
│  │   → ProtoRole: "做 X/Y/Z 类事的人" (置信度)           │
│  │                                                        │
│  ├─ 几个元素是否在语义上属于同一主题？                    │
│  │   → ProtoConcept: "似乎属于同一类" (置信度)           │
│  │                                                        │
│  └─ 整个场景是否在往某个方向推进？                        │
│      → ProtoPurpose: "这个地方/流程似乎在试图达成X"      │
│                                                        │
│  Step 2: 形成初始原型                                     │
│  ├─ 对每个 ProtoType 赋予:                                │
│  │  • 置信度（基于观察的充分度和一致性）                  │
│  │  • 反例列表（不符合这个原型的观察）                    │
│  │  • 待验证问题（需要更多证据的方面）                    │
│  └─ 关键: 始终用概率语言，而非确定语言                    │
│                                                        │
│  Step 3: 向用户呈交感知摘要                               │
│  ├─ "经过 3 次观察，我注意到:                              │
│  │  • 挂号→等待→被问 这个序列似乎比较稳定 (置信度 0.6)   │
│  │  • 穿白衣服的人有两种: 问问题的和做检查的 (0.45)      │
│  │  • 整个流程似乎是为了找出问题+解决问题 (0.35)         │
│  │                                                        │
│  │  我说得对吗？有什么明显的错误吗？"                      │
│  └─ 用户纠正 → 修正原型、调整置信度                       │
│                                                        │
│  输出: ProtoSequence[] + ProtoRole[] + ProtoConcept[]     │
│        + ProtoPurpose[] (每个带置信度)                    │
└─────────────────────────────────────────────────────────┘
```

### Phase 3: 互动验证（Interactive Validation）

**目标**：通过与世界的每一次互动，精化原型。

```
┌─────────────────────────────────────────────────────────┐
│            Interactive Validation                         │
│                                                           │
│  每次该场景再次出现时:                                    │
│                                                           │
│  Step 1: 激活当前最高置信度的原型                          │
│  ├─ 如果 ProtoSequence 置信度 > 0.5:                      │
│  │   → 尝试按序列引导行动                                 │
│  │   → "根据我观察到的模式，现在应该去 X"                 │
│  │   → 如果正确 → 置信度 +0.1                             │
│  │   → 如果错误 → 置信度 -0.2（错误比正确信息量更大）    │
│  │   → 记录: 什么条件下这个预测失败了                     │
│  └─ 如果置信度 < 0.5: 继续观察，不主动行动               │
│                                                           │
│  Step 2: 主动提问                                         │
│  ├─ 每个原型的"待验证问题"按优先级排序                    │
│  ├─ 在合适时机（遵守治理规则）向用户提问:                  │
│  │   "我注意到有些情况下去医院要先取号再挂号，             │
│  │    有些情况直接挂号。这取决于什么？"                    │
│  └─ 用户回答 → 原型增加新维度（"就诊类型影响流程"）      │
│                                                           │
│  Step 3: 搜索外部知识                                      │
│  ├─ 当原型置信度 > 0.6 且仍有待验证问题:                  │
│  │   → 主动搜索外部资源（browser/search）                │
│  │   → "医院就诊流程" → 找到标准流程描述                 │
│  │   → 与自有原型对比                                     │
│  │   → 差异 = 新的验证问题                                │
│  └─ 外部知识不替代自有经验——两者互相验证                  │
│                                                           │
│  Step 4: 原型合并与分裂                                    │
│  ├─ 当两个 ProtoRole 的职责边界越来越清晰:                │
│  │   → 正式分裂为两个独立的 ProtoRole                    │
│  ├─ 当两个 ProtoConcept 被发现是同一个东西的不同名称:     │
│  │   → 合并为一个 ProtoConcept                            │
│  └─ 记录合并/分裂历史（元数据: 什么时候、什么证据)       │
│                                                           │
│  输出: 持续更新的 ProtoStructure (置信度演变)              │
└─────────────────────────────────────────────────────────┘
```

### Phase 4: 固化 / 退化（Crystallize / Degrade）

**目标**：当原型置信度达标时，固化为可复用的认知结构。当固化结构遇到反例时，退化为原型。

```
┌─────────────────────────────────────────────────────────┐
│            Crystallize / Degrade                          │
│                                                           │
│  固化条件:                                                │
│  ├─ ProtoSequence 所有关键步骤置信度 > 0.8               │
│  ├─ ProtoRole 核心职责置信度 > 0.8                       │
│  ├─ ProtoPurpose 置信度 > 0.7                            │
│  └─ 观察次数 >= min_observations (默认 5)                │
│                                                           │
│  固化流程:                                                │
│  ├─ 生成 CandidateStructure (交给 V5 的 Registry)        │
│  ├─ 标记来源: "从 Proto-Cognitive Engine 演化而来"       │
│  ├─ 保留原型历史: 固化结构包含完整的演化链路             │
│  └─ 后续优化交给 V5 的 Structure Evolution 流程          │
│                                                           │
│  退化条件:                                                │
│  ├─ 固化后的结构在新经验中持续出现反例 (> 3 次)           │
│  ├─ 结构的预测准确率下降到 < 0.7                         │
│  └─ 用户主动表示"这个理解不对"                           │
│                                                           │
│  退化流程:                                                │
│  ├─ 标记结构为 "degraded"                                │
│  ├─ 回到 ProtoStructure 状态（保留已有的置信度数据）     │
│  ├─ 反例作为新的修正信号                                  │
│  ├─ 重新进入互动验证阶段                                  │
│  └─ 如果多次退化的根源相同 → 标记为 StructuralGap        │
│      → 交给 V5 的 Constructor 重新设计                   │
│                                                           │
│  关键: 固化不是终点。任何结构都可能在新证据面前退化为      │
│        原型，然后重新精化。这是"越管越好"的底层机制。     │
└─────────────────────────────────────────────────────────┘
```

## 层自修改流程

V6 允许 Praxis 在人类治理下修改自己的架构组织方式。

```
┌─────────────────────────────────────────────────────────┐
│            Layer Self-Modification Flow                    │
│                                                           │
│  Step 1: 检测架构级问题                                   │
│  ├─ Structural Inadequacy Detector 发现:                   │
│  │   某个认知结构的"形状"在现有层中无处安放               │
│  │   例如: "L3 存储知识, L4 学习知识, 但某类知识          │
│  │          学到即用——它应该属于 L3 还是 L4？"           │
│  └─ → 升级为 ArchitectureGap (不是普通的 StructuralGap)   │
│                                                           │
│  Step 2: 生成架构变更提案                                 │
│  ├─ 分析:                                                 │
│  │  • 受影响的范围（哪些场景、哪些层、哪些子系统）        │
│  │  • 变更的级别（Level 1-6）                             │
│  │  • 替代方案（不改层面改什么？）                        │
│  └─ 生成 ArchitectureChangeProposal                       │
│                                                           │
│  Step 3: 多级审批                                          │
│  ├─ Level 3-4: 用户 + 运维者双重审核                     │
│  ├─ Level 5: 同上 + 外部专家可选                          │
│  ├─ Level 6: 人类手动修改（Praxis 不能提议）             │
│  └─ 所有级别: 需要回滚方案                                │
│                                                           │
│  Step 4: 受控实施                                          │
│  ├─ 不是一次性替换 → 渐进迁移                             │
│  ├─ 新旧架构并行运行                                       │
│  ├─ 核心指标持续监控                                      │
│  └─ 退化 > 5% → 自动暂停                                  │
│                                                           │
│  Step 5: 版本管理                                          │
│  ├─ 架构版本号: major.minor.patch                         │
│  │  • patch: 参数调整 (Praxis 自主)                      │
│  │  • minor: 新结构增加/结构修改 (人类审核)               │
│  │  • major: 层定义/组织方式变化 (双重审核)               │
│  └─ 完整回滚路径: 任何一个版本都可以降级回去              │
└─────────────────────────────────────────────────────────┘
```

## V6 完整演进示例：从零到"医院专家"

```
第 1 次去医院:
  Open Perception → 标记 15 个 SalientElement
  → 生成 5 个感知性问题（"挂号是什么？""医生和护士的区别？"）
  → ProtoSequence 初始置信度 0.25

第 2 次去医院:
  上次的原型被激活 → 5 个预测 → 3 个正确 +1, 2 个错误 -2
  → ProtoSequence 置信度 0.25 → 0.35
  → ProtoRole "医生" 置信度 0.3 → 0.5
  → 新问题: "为什么这次先取号？和上次直接挂号不一样"

第 3 次去医院:
  发现"分诊"这个新概念
  → ProtoSequence 增加了"分诊"步骤
  → 新 ProtoConcept: "就诊类型"（急诊 vs 门诊）
  → ProtoSequence 分裂为两个变体

第 5 次去医院:
  ProtoSequence "门诊流程" 核心步骤置信度 > 0.8
  → 提出固化: ProcessTemplate "医院门诊就诊流程"
  → 用户审核 → 批准

第 8 次去医院:
  遇到"住院"这个分支
  → ProcessTemplate "门诊流程" 在住院场景中不适配
  → 退化: 模板标记为 "门诊限定"
  → 开启新的 ProtoStructure "住院流程"

...

6 个月后:
  Cognitive Structure Registry 中:
  • HospitalVisit.门诊流程 v2.1 (crystallized, 23 次验证)
  • HospitalVisit.急诊流程 v1.8 (crystallized, 12 次验证)
  • HospitalVisit.住院流程 v1.2 (crystallized, 5 次验证)
  • DomainKnowledge "医疗健康" familiarity: 0.75
  • Role "医生" v3 / Role "护士" v2 / Role "药剂师" v1
  
  Praxis 从"医院是什么都不知道"变成了"医院就诊专家"。
  没有人类为它设计任何一个模板——它自己观察、自己构造、自己精化。
  人类做的只是回答它的感知性问题、纠正它的错误假设、审核它的固化提案。
```
