# Where does Praxis V6 sit?

## 架构定位（V6：Proto-Cognitive Engine + 层自修改）

V6 在 V5 的 Meta Layer 基础上，新增 **Proto-Cognitive Engine**——它是 Meta Layer 中的一个新子系统，但它处理的不是"结构匹配失败"（V5），而是"结构根本不存在"。

更重要的是，V6 打破了 V1-V5 的一个隐含假设：**架构的结构（层定义、层边界、子系统组织）是固定的**。

## Proto-Cognitive Engine 的位置

它位于 Meta Layer 内部，但与 V5 的三个子系统有明确分工：

```
Meta Layer:
  │
  ├─ Structural Inadequacy Detector (V5)
  │   触发条件: 模板匹配失败（适配度 > 0 但 < 0.3）
  │   处理: Known-Unknown 结构缺口
  │   → 输出 StructuralGap → Constructor → 候选结构
  │
  └─ Proto-Cognitive Engine (V6)
      触发条件: 适配度 = 0（零先验）
      处理: Unknown-Unknown——连这是什么类型的场景都不知道
      → 开放感知 → ProtoStructure → 互动精化 → 候选固化
```

## 层自修改：架构不再固定

V1-V5 的架构是稳定的：6 层（V1）+ 4 子系统（V4）+ Meta Layer（V5）= 固定的组织结构。这是人类设计的。

V6 允许 Praxis 在人类治理下修改这个组织结构：

```
可能发生的修改场景:

场景 A: 合并层
  Praxis 在处理某类场景时发现 L3（知识管理）和 L4（学习闭环）
  在这个场景中处理的知识是"过程性知识"——学到的同时就要用。
  存储和学习的边界模糊了。
  
  → 提议: 在这个场景中创建 L3.5 "过程知识引擎"
  → 它结合了 L3 的存储和 L4 的学习，但只针对过程性知识
  → 不影响 L3 和 L4 在其他场景中的独立运作

场景 B: 替代层级
  Praxis 在处理"创意探索"场景时发现层级结构根本不适配——
  创意过程是网络状的、非线性的、反复跳跃的。
  
  → 提议: 对创意类场景，使用 NetworkCognitiveModel 替代层级结构
  → NetworkCognitiveModel: 节点（想法）+ 边（联想）+ 激活传播
  → 层级结构在非创意场景中继续使用

场景 C: Meta Layer 自身修改
  Praxis 发现 Structural Inadequacy Detector 的检测逻辑
  在"情感密集型场景"中有盲区——它关注流程效率，但不关注情感适配。
  
  → 提议: 给 Detector 增加"情感适配质量"检测维度
  → 只影响检测逻辑，不影响其他子系统
  → 旧版 Detector 并行运行 2 周作为对照
```

## 架构稳定性治理

层可以变，但稳定性必须保证：

```
┌─────────────────────────────────────────────────────────┐
│           Architecture Stability Governance              │
│                                                           │
│  不可变（Immutable）:                                     │
│  • GovernancePolicy 本身（只能人类手动修改）               │
│  • HumanApprovalRequired 规则（不能被降级或删除）         │
│  • 回滚机制（任何修改必须有可执行的回滚路径）             │
│                                                           │
│  高频可变（Mutable - Fast）:                              │
│  • CognitiveStructure 的创建和参数调整                    │
│  • ProtoStructure 的形成和精化                            │
│                                                           │
│  低频可变（Mutable - Slow）:                              │
│  • 层定义和层边界的修改（每月最多 1 次）                  │
│  • Meta Layer 自身的修改（每季度最多 1 次）               │
│  • 架构组织方式的替换（每条新方式至少验证 3 个月）        │
│                                                           │
│  退化护栏（Regression Guardrails）:                       │
│  • 任何架构修改必须在所有已知场景上通过回归测试           │
│  • 如果核心指标退化 > 5%，自动暂停 + 通知人类            │
│  • "核心指标"由人类定义（不可被 Praxis 修改）           │
└─────────────────────────────────────────────────────────┘
```
