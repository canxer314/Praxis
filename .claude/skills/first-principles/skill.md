# /first-principles Command

用第一性原理方法分析一个问题或主张，区分事实、假设和类比，从基本真理出发重新推导。

## Behavior

### Step 0: 加载插件规则

读取 `.claude/skills/first-principles/rules.md`，将 8 条推理规则注入当前会话上下文。如 rules.md 不存在，使用本文件内置的默认规则。

### Step 1: 澄清分析对象

向用户确认：
- 要分析的具体问题或主张是什么？
- 分析的目的是什么（做决策？理解原理？挑战假设？创新设计？）
- 分析的深度要求（quick: 30 秒快速区分事实/假设；deep: 完整的 5 层追问）

如用户输入模糊（如"分析一下电动车"），追问具体主张（"电动车是否会在 5 年内取代燃油车？"vs"电动车的核心技术瓶颈是什么？"）。

### Step 2: 列出基本约束

识别该问题的**不可否认的约束**：

| 约束类型 | 示例 |
|---------|------|
| 物理约束 | 能量守恒、材料极限、光速 |
| 逻辑约束 | 数学定理、信息论边界 |
| 经济约束 | 边际成本不为零、稀缺性 |
| 人的约束 | 认知偏差、有限注意力、激励结构 |

这些约束是推理的锚点——任何违反它们的结论都可以直接排除。

### Step 3: 分解问题

将问题拆解为组件，对每个组件追问：
- 这个组件的功能是什么？（不是"它现在怎么实现的"，而是"它要实现什么"）
- 实现这个功能的最低必要条件是什么？
- 现有方案中的哪些部分是必要的，哪些是历史的偶然？

### Step 4: 剥离类比

扫描用户描述和已有分析中是否有类比思维：
- "就像 Uber 一样" → Uber 模式是历史偶然，不是物理约束
- "行业标准做法是" → 描述可以保留，但不能作为论证依据
- "大家都在用 X" → 标记为 `[类比/惯例，非论证]`

### Step 5: 重新构建

从基本约束出发，推导可能方案空间：
1. 从约束推导出"必须满足的条件"
2. 从条件展开"可能的实现路径"（不参考已有方案）
3. 比较可能的实现路径与现有方案
4. 识别现有方案中哪些是"因为历史原因而非物理原因"存在的

### Step 6: 输出结构化分析

```markdown
## 分析：[主题]

### 基本约束
{Step 2 列出的不可否认约束}

### 问题分解
{Step 3 的组件拆解}

### 类比剥离
{发现的类比思维及其替代视角}

### 从第一性原理推导
{Step 5 的推导过程}

### 结论
{核心发现，每条附带可验证预测}

### 与现有方案对比
{当前主流做法 vs 从第一性原理推导出的可能方案}
```

## 失败处理

| 场景 | 处理 |
|------|------|
| **用户问题无法找到基本约束** | 诚实说明"该领域缺少可用的基本约束"，建议用其他分析框架（如类比分析、专家意见聚合） |
| **分析对象过于宽泛** | 不强行分析，引导用户缩小范围："这个问题太宽了，能聚焦到一个具体主张吗？" |
| **用户期望快速回答但问题适合深度分析** | 告知 tradeoff，建议标记为 /task 后续深度分析 |

## 反模式

| # | 反模式 | 正确做法 |
|---|--------|---------|
| 1 | 跳过基本约束直接给方案 | 先列出物理/逻辑/经济约束，再推导 |
| 2 | 把行业惯例当作论证 | 惯例可以描述，不能作为"所以是对的"的理由 |
| 3 | 只问 1-2 层"为什么"就停下 | 追问直到用户说"我不知道"或到达不可分解的基本事实 |
| 4 | 结论没有可验证预测 | 每条结论必须附带"如果对/错，应观察到什么" |
