# What is AgentOS V4?

> V4 = V3 的多维主动学习 + **过程模型** = 一个知道"下一步该干什么"的 AgentOS

## 一句话定义

**AgentOS V4 在 V3 的"多维主动学习者"之上，增加了一套过程模型——让它不只是"会学习"，而是"会推进"：知道一件事情应该怎么走、现在走到哪了、下一步该干什么、该找谁、卡住了怎么办——像一个真正的秘书或程序员一样，不是等着被告诉每一步，而是自己驱动过程向前。**

---

## Is / Is-not

| Is | Is-not |
|----|--------|
| 过程状态的管理者（追踪 + 推动） | 过程的执行者（执行还是 OpenClaw Agent Loop） |
| 知道"下一步该干什么"的决策引擎 | 知道"下一步怎么干"的执行引擎 |
| 协作者关系与状态的维护者 | 协作者本人（不替代 PM/架构师/客户） |
| 阻塞检测 + 推动策略的决策者 | 无原则的催促工具（推动有策略、有节奏） |
| 行动正确与否的验证者 | 行动的发起者（发起由 Process Engine 触发，执行由 Agent Loop 完成） |

---

## V1 → V2 → V3 → V4 演进

| 维度 | V1 | V2 | V3 | V4 |
|------|----|----|----|----|
| **核心命题** | AI 需要跨会话成长 | 用什么承载成长 | 学什么、谁发起 | **怎么把学到的变成正确的下一步** |
| **任务模型** | 工具链 | 工具链 | 工具链 | **过程网络**（多角色 × 多步骤 × 多依赖） |
| **角色认知** | 只有"用户" | 只有"用户" | 用户模型（单角色） | **角色模型**（多角色：用户+PM+架构师+...） |
| **执行模式** | AgentOS 自己做 | 观察 OpenClaw 做 | 观察 + 主动学习 | **观察 + 推动 + 等待 + 升级** |
| **阻塞处理** | 无 | 无 | 无 | **阻塞检测 + 推动策略 + 升级路径** |
| **能力维度** | 1 维 | 1 维 | 4 维 | **6 维**（+过程管理 + 行动验证） |
| **学习验证** | 工具成功/失败 | 同 V1 | 同 V1 + 知识缺口 | **+ 行动正确性验证 + 流程改进** |

## 两个具体场景，同一个核心能力

### 场景 A：秘书（流程中遇到信息）

> 用户通过 Telegram 发来一份 PDF 合同，没有说任何话。

**V3 的反应**：`message_received` → "收到一个文件" → 意图分类："文件上传" → ❓ 然后呢？

**V4 的反应**：
1. Process Engine 识别到"文件摄入流程"启动
2. 当前步骤："分类 + 关联 + 提取"
3. OCR 提取 → 内容："膜力云项目合同，付款周期 60 天..."
4. 分类决策 → 领域=膜力云，类型=合同
5. 关联检测 → "关联到膜力云项目的 3 份已有文件"
6. **矛盾检测** → "⚠️ 5 月 8 日会议纪要写付款周期 30 天，这份合同写 60 天"
7. Momentum Engine 决策：中等优先级矛盾，不紧急但有价值 → 主动提醒用户
8. 用户回复"以合同为准" → Action Verification：矛盾检测有价值 ✅ → 下次更积极检测类似矛盾

### 场景 B：程序员（流程中遇到人）

> 用户："把用户管理模块做了"

**V3 的反应**：TaskPattern 匹配不到 → 开始写代码 ← **跳过了需求、设计、排期！**

**V4 的反应**：
1. Process Engine 识别到"软件开发流程"模板
2. 当前阶段："需求分析"，下一步："与 PM 确认 PRD"
3. Role Model 查 PM 张三：沟通偏好=飞书，可用时间=周二/周四下午，当前负载=高
4. Momentum Engine 生成沟通草稿 → 发给 PM → 设置等待超时 3 天
5. 3 天后 PM 没回 → 阻塞超时 → 催办（非升级）
6. 7 天后还没回 → 升级：通知用户"PM 张三 7 天未确认 PRD，建议您直接沟通"
7. PM 回复了 → 出口条件满足 → Process Engine 进入下一阶段"架构设计"
8. 同理推进：架构师 → 开发经理 → 写代码 → 测试 → 验收 → 闭环

## V4 的核心洞察

秘书场景和程序员场景看起来完全不同，但本质是**同一个问题的两种表现**：

```
V1-V3 的任务模型:
  用户给任务 → AgentOS 做 → 交付用户
  （AgentOS 是执行者，一条工具链）

V4 的任务模型:
  用户给目标 → AgentOS 推动过程 →
  ┌─────────────────────────────────────┐
  │  步骤1       步骤2       步骤3      │
  │  找A做X  →  等B确认  →  找C沟通    │
  │  (自己做)    (等待)     (催办)      │
  │   ↓ 阻塞     ↓ 超时     ↓ 升级     │
  │  推动/绕过   升级       通知用户    │
  └─────────────────────────────────────┘
  → 交付用户
  （AgentOS 是推动者，一张协作网络）
```

秘书场景卡在**步骤内**的信息处理（怎么摄入 PDF、怎么提取结构、怎么检测矛盾）。
程序员场景卡在**步骤间**的角色协调（该找谁了、怎么催、催不动怎么办）。

但两者都需要同一个东西：**知道自己现在在哪、下一步该干什么、卡住了怎么推动**。这就是 V4 要给的。
