# Why Praxis V4?

> V3 让 Praxis 知道了"学什么"和"自己缺什么"。但它不知道"下一步该干什么"。
> V4 解决：学到的东西怎么变成正确的下一步行动——尤其是下一步不是自己做，而是找别人做的时候。

---

## 一、根本问题回顾（从 V1 到 V3 的演进链）

```
V1: LLM 无状态 → 需要跨会话持久学习
V2: 学习闭环需要载体 → OpenClaw Memory Plugin + AgentMemory
V3: 学习不能只是工具熟练度 → 四维能力模型 + 主动好奇心
V4: 学到的知识需要驱动行动 → 过程模型 + 角色协调 + 推动引擎
```

每一版解决上一版暴露的根本缺陷。V4 要解决的缺陷是：**V3 学会了很多东西，但它不知道在什么时候用哪一样来驱动下一步行动。**

---

## 二、V3 的任务模型有什么问题？5 Whys

### 场景：用户说"把用户管理模块做了"

**Why 1**：为什么 V3 不会先去找 PM 聊需求？
→ 因为 V3 的 TaskPattern 是**工具链**——它理解"做 PPT = data_fetch → chart → ppt_gen → send"。但它不理解"做模块 = 找 PM → 找架构师 → 找开发经理 → 写代码 → 找测试 → 验收"。后者不是工具链，是**协作链**。

**Why 2**：为什么 V3 把一切任务建模为工具链？
→ 因为 V1 的原始假设是"Praxis 是独立 Harness，所有任务都可以分解为自己调用工具"。这个假设贯穿了 V2（迁移）和 V3（扩展学习），从未被问题化。

**Why 3**：为什么这个假设从未被问题化？
→ [假设] 因为前三轮设计的讨论场景都是"一个人 + AI"的闭环——用户给任务，AI 执行，用户反馈。没有引入"AI 需要去找其他人协调"的场景。这种场景在工程现实中极其常见，但在设计中未被覆盖。

**Why 4**：在多人协作场景中，Praxis 最缺的是什么能力？
→ **过程感知**。三个不可约简的缺失：
1. 不知道"事情应该怎么走"——没有流程模板（软件开发流程是什么）
2. 不知道"事情现在走到哪了"——没有过程状态（需求阶段？设计阶段？卡在哪个步骤？）
3. 不知道"怎么推动事情往前走"——没有推动策略（找谁？怎么说？卡住了怎么办？）

**Why 5**：为什么这三个能力是 Praxis 的职责，不是 OpenClaw Agent Loop 的职责？
→ 因为：
1. 流程模板需要跨会话持久存储——Agent Loop 是无状态的
2. 过程状态跨会话——这次会话找了 PM，下次会话 PM 回复了，Praxis 必须记得中间经过了多少天
3. 推动策略需要学习——上次催 PM 的方式有效吗？哪种升级路径成功率最高？这需要跨会话的数据积累
4. 角色模型需要持久化——PM 张三的沟通偏好不会变，Praxis 应该记住而不是每次都问

## 三、两个看似不同的场景，同一个根因

### 秘书场景（信息管理）暴露的问题

| 步骤 | 需要什么能力 | V3 有没有 |
|------|------------|----------|
| 收到 PDF 合同 | 触发文件摄入流程 | ❌ 没有文件摄入流程模板 |
| OCR + 提取内容 | 工具调用（有） | ✅ |
| 判断这个文件属于哪个项目 | 分类决策 + 领域匹配 | 部分（有领域知识，但没有分类流程） |
| 发现与已有信息的矛盾 | 矛盾检测 + 关联验证 | ❌ 没有主动矛盾检测 |
| 提醒用户矛盾存在 | 推动策略：中等问题 vs 紧急问题 | ❌ 没有推动分级 |
| 用户确认后更新知识 | 知识更新 + 行动验证 | ✅（知识更新有）+ ❌（行动验证没有） |

### 程序员场景（角色协调）暴露的问题

| 步骤 | 需要什么能力 | V3 有没有 |
|------|------------|----------|
| 知道下一个步骤是"找 PM 聊需求" | 流程模板 + 阶段门控 | ❌ |
| 知道 PM 是谁、怎么沟通 | 角色模型（多角色） | ❌（只有用户模型） |
| 生成给 PM 的沟通草稿 | LLM + 角色适配 | ✅（LLM 可以做） |
| 等 PM 回复 → 超时 → 催办 | 阻塞检测 + 超时策略 | ❌ |
| 催了两次没反应 → 升级给用户 | 升级路径 + 阈值管理 | ❌ |
| PM 确认后进入下一阶段 | 阶段门控 + 下一步触发 | ❌ |

### 统一的根因

两个场景的每一个 ❌ 都指向同一个缺失：**V3 没有建模"过程"**。

过程不是工具链。过程包含：
- **步骤的先后依赖**（做完 A 才能做 B）
- **步骤的承担者**（A 是 Praxis 自己做，B 是找 PM 做）
- **步骤的入口/出口条件**（什么条件下可以进入/退出这一步）
- **步骤的超时和阻塞处理**（等多久？等不到怎么办？）
- **步骤间的状态传递**（上一步的结果如何影响下一步）

## 四、V4 需要解决的核心问题

### 问题一：过程建模

V3 只有 `TaskTrace`（事后记录），没有 `ProcessTemplate`（事前模板）和 `ProcessInstance`（事中状态）。

V4 需要让 Praxis 能够在三种模式之间切换：

```
模式 A: 自己做（V3 已有的工具链模式）
  冲咖啡: status() → grind() → brew() → pour()

模式 B: 找人做（V4 新增的协作模式）
  确认需求: 起草PRD → 发给PM → 等待反馈 → 确认/修改循环 → 出口

模式 C: 混合（V4 新增）
  做周报: 采集数据(自己做) → 生成图表(自己做) → 发PM审阅(找人做) → 修改(自己做) → 发送(自己做)
```

### 问题二：角色建模

V3 只有一个 `UserModel`。V4 需要 `RoleModel`——一个角色注册表，每个角色有：

- 身份：姓名、角色、职责
- 沟通：偏好渠道、可用时间、回复速度
- 关系：与用户的关系、与项目的关系、与其他角色的关系
- 状态：当前负载、当前情绪倾向、最近互动时间
- 历史：互动记录、响应率、沟通效果评分

### 问题三：推动策略

当流程卡住时，Praxis 需要判断：

```
┌─────────────────────────────────────────┐
│          推动策略决策树                   │
│                                          │
│  流程卡在步骤 S，等待角色 R，已等 T 天   │
│                                          │
│  T < 合理等待期?                         │
│    → 继续等待，不行动                     │
│                                          │
│  T > 合理等待期 且 T < 升级阈值?          │
│    → 催办（轻度/中度）                    │
│    → 通过 R 的偏好渠道                    │
│    → 语气适配 R 的沟通风格                │
│                                          │
│  T > 升级阈值?                            │
│    → 升级通知用户                         │
│    → 提供上下文：等了多久、催了几次        │
│    → 建议：用户直接沟通 / 找 R 的上级      │
│                                          │
│  R 明确拒绝?                              │
│    → 通知用户                             │
│    → 建议替代方案（换人/绕过/重新定义需求）│
└─────────────────────────────────────────┘
```

推动策略**不是写死的**——Praxis 从历史互动中学习：
- PM 张三催一次通常就回 → 合理等待期可以设长一点
- 架构师李四催三次才回 → 合理等待期设短，提前开始催
- 客户王五催了反而反感 → 对王五只升级不催办

### 问题四：行动验证

V3 验证的是"工具调用是否成功"和"知识缺口是否存在"。V4 需要验证的是"我做的这个推动动作对吗"：

```yaml
# V4 新增的验证维度
ActionVerification:
  - 我选择找 PM 聊需求 → 对吗？还是应该先找客户？
  - 我等了 3 天才催 → 等太久了吗？还是催太早了？
  - 我用飞书而不是邮件联系 PM → 对吗？
  - 我催办的语气 → 太软了还是太硬了？
  - 这次流程总共花了 12 天 → 哪些步骤可以并行？
```

每次流程结束后，Praxis 反思这些维度，更新：
- `ProcessTemplate`：这个流程的标准步骤是否需要调整
- `RoleModel`：对这个角色的沟通策略是否需要调整
- `MomentumStrategy`：推动时机和方式是否需要调整

## 五、反向论证

**反方论点**："Praxis 不需要过程模型——过程协调是 OpenClaw Agent Loop 的职责。LLM 本身就能推理出'下一步应该找 PM'，Praxis 只需要在 system prompt 里注入相关上下文就行。"

**为什么这个论点不够**：

1. **LLM 不知道谁是谁** — system prompt 里可以注入"PM 是张三"，但"张三上周催了三次没回、这周他比较忙、上次用飞书联系效果比邮件好"——这些信息需要跨会话积累，只有 Praxis 有。
2. **LLM 不知道流程走到哪了** — 上次会话找了 PM，这次会话 PM 回了。这两次会话之间隔了 4 天。LLM 的无状态窗口看不到"等了 4 天"这个事实——Praxis 的 ProcessInstance 里有。
3. **LLM 不会自己触发** — 用户不发消息，LLM 不动。但"等了 7 天该升级了"这件事需要 Praxis 通过 cron 或 session 恢复时主动触发，不能等用户想起来。
4. **LLM 不会从推动经验中学习** — 催了 10 次 PM，有的方式有效有的方式无效。LLM 每次都从零开始推理。Praxis 应该记住"对张三，催一次就够了；对李四，催三次不如直接升级给用户"。

**什么条件会改变结论？**
- 如果任务永远不需要其他人参与（纯单人 AI 助手）→ V3 足够
- 如果流程永远在一轮对话内完成（不需要等待和跨会话）→ Agent Loop 能处理
- 如果每个协作者的行为完全一致（不需要学习沟通策略）→ 不需要 RoleModel

但在现实的工作场景中，以上三个条件没有一个成立。

## 六、V4 的成本与取舍

| V4 新增的复杂度 | 带来的收益 |
|---------------|-----------|
| ProcessTemplate 需要预定义或学习 | Praxis 不再跳步骤（写代码之前先确认需求） |
| RoleModel 需要维护 | Praxis 知道找谁、怎么找、什么时机找 |
| MomentumEngine 需要调参 | 不烦人、不沉默、在正确的时间做正确的推动 |
| ActionVerification 需要事后反思 | "越管越好"——第 10 次流程显著快于第 1 次 |

**最大的风险**：推动过度。一个过于激进的 Momentum Engine 会让用户和协作者都觉得烦。缓解措施：
- 首次部署默认为保守策略（长等待期、先提醒用户、不直接催办协作者）
- 用户可随时调整任何协作者的推动策略
- 每次升级或催办都会征求用户确认（至少在初期）

**明确的边界**：V4 的 Process Engine **不执行步骤**——它追踪和推动，但不替代执行。写代码的是 OpenClaw + code_editor 工具，审核代码的是架构师，测试的是测试人员。Praxis 只是确保每一步在对的时候发生。
