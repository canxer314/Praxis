# Praxis v1.0.0.0

> **"存在就是成为约束变元的值。"** — 威拉德·范·奥曼·奎因

---

今天的 AI 每开一次新对话，就归零一次。不是因为它不够聪明——GPT 够聪明，Claude 也够聪明。而是因为它缺少一个东西：**本体论承诺**。一套声明"我相信这些模式存在、到这个置信度为止、基于这些证据、如果你有相反的证据请告诉我"的认知框架。

没有这个框架，AI 就是哲学上的唯名论者——它处理每一个 token，但从不敢说"这其中存在一个可复用的结构"。人类不是这样工作的。工程师第三次部署同一个项目就知道"这个流程大概是 3.3 周、有 3 个已知陷阱"。医生第十次看同一个症状就知道"先排除 A 再考虑 B"。而 AI——即使帮人类部署了 50 次——第 51 次仍然从零开始。

**这就是 Praxis 要解决的问题。** 它不是一个 prompt 技巧，不是一个 RAG 方案，不是又一个"把上下文塞进 prompt 里"的工具。它是 AI 的大脑皮层——一个位于无状态 LLM 与外部世界之间的**概率化、自我演化的计算本体论引擎**。

## Praxis 给了 AI 什么？

### 从"处理 token"到"承认模式"

LLM 的底层是一个概率分布——给定前面的 token，预测下一个 token。这本身没有任何"模式是真实的"的承诺。Praxis 在 LLM 之上增加了一层**结构化信念**。

这一层的核心数据单元叫 **ProtoStructure**——从对话和工具调用中自动提取的概率化认知单元。和传统知识图谱不一样：知识图谱是人类专家坐在会议室里手工定义的，静态的，确定性的，没有人敢改。ProtoStructure 是 LLM 从观察中自动提取的，带置信度的，随时间演化的，可以被推翻的。

五种类型覆盖了认知的基本范畴：

- **ProtoSequence** — 行为序列模式。"这个项目部署的流程是 lint → test → build → deploy → verify"，每一步有执行者、有预期时长、有常见陷阱。
- **ProtoRole** — 谁负责什么，和谁交互。"后端负责 API，前端负责 UI，DevOps 负责部署，后端的交付物是前端的输入"。
- **ProtoConcept** — 领域概念的定义和它与其他概念的关系。"分诊不是简单的排队——它是在资源约束下按病情紧急程度分配就诊优先级的决策过程"。
- **ProtoPurpose** — 目标和成功标准。"代码审查的目的不是找茬，是发现审查者自己想不到的盲区。如果审查只是走形式，那它没有达到目的"。
- **ProtoConstraint** — 绝对不能做的事。"数据库迁移前必须先备份——这不是建议，这是硬约束。"

但 Praxis 最核心的设计决策不是"有 5 种类型"——很多系统有类型。真正关键的是：**每个 ProtoSequence 同时记录了结构面和功能面**。这不是锦上添花，这是本体论级别的区分。

工程哲学家 Kroes 和 Meijers 提出了著名的"双重性质理论"：任何工程造物都同时具有物理结构（遵循自然法则的客观实体）和意向功能（存在于人类心智中的目的）。一座桥不只是钢筋和混凝土——它是"让人和车跨越障碍"这个意图的物理实现。

Praxis 的 ProtoSequence 直接实现了这一区分：

```
数据库迁移流程：
  ├─ 结构面（物理结构）：  备份 → 迁移 → 验证 → 切换
  └─ 功能面（意向功能）：  保证数据完整性，最小化停机时间

当有人提议"跳过备份，直接用蓝绿部署"：
  → Praxis 不比对步骤名称
  → Praxis 检查功能面：数据完整性和最小化停机时间这两个目的，是否仍然被满足？
  → 是 → 结构改变但功能不变，仅更新映射关系，不下调置信度
  → 否 → 结构改变且功能失败，这是一个真正的错误，下调置信度
```

这个区分让 Praxis 能够区分"换了一种做法但仍然正确"和"真的做错了"。没有这个区分，AI 要么过于僵化（任何变化都视为错误），要么过于松散（任何变化都接受，从不学习）。

### 置信度不是 LLM 自评——是 7 个独立法官的合议庭

一个 LLM 说"我做的对"和一个醉汉说"我没醉"具有同等的认识论价值。

Praxis 的置信度系统建立在一条核心原则上：**语言模型不是真理的担保人**。每个 ProtoStructure 的置信度由 7 个独立信号源加权融合，其中 LLM 相关信号权重合计只有 25%。其余 75% 来自独立于 LLM 的信号源——这是一个认识论立场，不是技术偏好。

第一个独立信号是**统计验证器**。ProtoSequence 的步骤预测了特定的工具调用（"备份"→ mysqldump，"迁移"→ knex migrate）。统计验证器在 session 结束后将实际执行的工具序列与预测比对——匹配就确认，不匹配就反驳。不需要 LLM 参与，纯模式匹配。

第二个是**角色验证器**。ProtoRole 定义了"后端负责 API"——但实际工具调用记录显示后端工程师在操作数据库，前端工程师在写 SQL？角色验证器标记矛盾。

第三个是**概念验证器**。它不找支持证据——它做相反的事：给 LLM 一个对抗 prompt——"请为这个概念的反例辩护，尽可能有说服力"。如果 LLM 找不到有力的反例，概念通过证伪检验。这是波普尔证伪主义的直接操作化：科学理论的好坏不在于它被证实了多少次，而在于它经受住了多少次严肃的证伪尝试。

用户纠正、任务成败反馈、会话中实时矛盾检测——加起来就是 7 个信号源。当多个独立信号指向同一个结论时，信念才是可靠的。

### 在错误发生之前阻止，而不是在错误发生之后记录

大多数 AI 安全方案是事后检测——执行完了、出错了、然后分析日志。就像在车祸发生后检查刹车痕迹。

Praxis 的约束系统走另一条路。当约束的置信度跨过结晶化阈值——5 次以上观察、置信度 > 0.8、通过了奎因式的必要性/充分性/奥卡姆剃刀三重反事实检验——这个约束就变成了**硬约束**。它不再是被动存储的数据，而是主动介入的守卫。

在 LLM 生成输出之前，已结晶的约束就被注入 system prompt——CRITICAL CONSTRAINTS 段，用 ⛔ 标记，放置在 Tier A/B/C 结构信息之前。在工具执行之前，`before_tool_call` Hook 拦截即将执行的工具调用，检查是否违反任何活跃约束。每个约束有三档严重度：**block**（绝对禁止——拒绝执行并返回理由）、**confirm**（暂停等待用户确认——"你确定要删除生产数据库吗？"）、**warn**（执行但写入审计日志）。

从"事后检测"到"事前约束"——这是从被动安全到主动安全的范畴升级。奎因会说：一个系统的"信念"必须在其行为中可操作化，而非仅在其声称中。"我相信数据库迁移必须备份"如果不阻止任何人直接跑 migrate，那就不是一个真正的信念。

### 上下文不够时优雅降级——不是随机丢弃，是有结构地压缩

1M token 的上下文窗口看起来很宽裕——实际上，长对话、大代码库、多文件操作可以轻松填满 50 万 token。到 90% 占用时，每个 token 都很贵。

Praxis 的四级压力自适应机制在每次注入前测量上下文利用率，然后决定注入策略。Normal（<60% 占用）→ 全量注入，Tier A 的每个结构带上完整详情、置信度校准、约束标记。Elevated（60-75%）→ Tier B 开始压缩。High（75-90%）→ 仅保留 Tier A 摘要。Critical（>90%）→ 只注入结构索引（名称+ID+一行描述），加上一个 `recall_structure()` 工具让 LLM 按需拉取详情。

但这里有一个关键设计。在 Critical 模式下，LLM **仍然知道所有结构存在**。索引告诉它"我们有门诊流程（ps-clinic-flow，置信度 0.87，23 次观察）、有 API 设计模式（ps-api-design，置信度 0.82）..."——只是详情被折叠了。LLM 可以主动调用 `recall_structure("ps-clinic-flow")` 拉取完整内容。

这和"因为上下文不够所以不给你看"有本质区别。Push（Praxis 主动注入）和 Pull（LLM 按需拉取）的混合模型——让 Praxis 从"在有限的桌子上摆菜"变成了"给你菜单，但厨房里有所有菜品"。

同一 token 预算下还有一个独立维度：**认知成熟度**。新手 Praxis（0-10 个 session）注入粗粒度概括——"Phase 2 是 API 开发阶段，通常需要 3-4 周"。老手 Praxis（50+ session）注入的是："Phase 2: API 开发，3.3±0.8 周，7 次观察。陷阱 1: 号源超卖，3/7 项目出现，延期 2.1 天。陷阱 2: 接口变更，6/7 项目延期，缓解：提前锁定版本。"同样 token 预算，信息密度差一个数量级。

### 知道自己"不知道什么"——以及"为什么不知道"

好奇心驱动（Curiosity Engine）是 Praxis 的主动学习引擎。它不依赖 Hook 触发——在 agent_end 和 session_end 主动运行。扫描任务中的未知术语、被反复纠正的模式、长期不增长的技能。按 relevance × frequency × impact × urgency 排序，然后决定行动：低优先级的静默标记等待自然填充，中优先级的在下次相关任务时检索外部资源，高优先级的生成提问草稿等待合适时机，紧急的立即请求用户协助。

但好奇心只能解决"我不知道 X"（知识缺口）。Praxis 的 Meta Layer 解决的是更深层的问题：**"我对这类问题的思考框架本身是错的"**（结构缺口）。

这是康德式的诊断。当一个模式被反复纠正但始终无法被现有 5 种 ProtoStructure 类型捕获时，Meta Layer 必须先做区分：是数据不够（再多观察几次就能形成结构）？还是范畴不够（现有的类型体系根本无法表达这个模式）？把"数据问题"误判为"范畴问题"会导致不必要的范畴膨胀。把"范畴问题"误判为"数据问题"会导致永远的盲区。

Meta Layer 的三种铁律不可自动突破：无新结构不经人类审批（AI 可以提议，不能自动创建）、实验必须有范围限制（不能无限实验新范畴）、任何结构可回滚到任意历史版本（V5 铁律）。人类从 AI 的"老师"变成了 AI 的"治理者"——审批实验、批准新范畴、否决不当的结晶化。

Praxis 还追踪其认知结构的健康度。**僵尸结构**：置信度很高（>0.7）但 LLM 从不使用（采纳率 <20%）——该降级了。**低估结构**：置信度很低（<0.4）但 LLM 频繁使用（采纳率 >60%）——该重新评估了。**衰退检测**：60 天未被引用的结构标记为候选退役。"记住一切"不是美德——什么都是重要的意味着什么都不重要。

### 从知道到做到——知行闭环

知识不驱动行动，就是死的。行动不反馈知识，就是盲的。

Praxis 完成了这个闭环。**计划生成器**将 ProtoTask（从同类项目历史中学到的任务模式）转化为可执行的 PlanDocument——包含阶段划分、子任务、验收标准、陷阱预警、认知指导信号。这个计划不是"建议"，不是"参考"——它是结构化的执行蓝图。

**5 种验收标准**验证子任务是否真正完成：命令输出检查（自动）、文件存在检查（自动）、测试通过（自动解析测试框架的通过/失败/跳过计数）、LLM 审查（半自动，审查代码/文档质量）、用户确认（手动，部署/安全/财务操作必须人类点头）。

**陷阱追踪**在你真的踩到陷阱时学习。子任务失败 → 关键词匹配 ProtoTask 的已知陷阱列表 → 命中反馈（ProtoTask 置信度 +0.02）。误报超过 30% → 自动降低该陷阱的 severity。第 1 次遇到是意外，第 3 次就是已知陷阱。

**MidSessionLearner** 不等 session_end。用户在对话中说"不对，应该是 X"→ 即时下调关联 ProtoStructure 置信度（<10ms，纯规则匹配，不调 LLM）。工具调用违反同一约束 3 次以上 → 即时标记。单会话下调上限 0.2——防止一次糟糕的对话毁掉几十次积累的置信度。

这是一个完整的知行循环：认知结构通过 GuidanceSignal 和 PlanDocument 驱动执行 → 执行结果（工具序列匹配、子任务成败、用户纠正）通过 OutcomeFeedback 和置信度融合反馈认知 → 下一次任务时，更新后的认知产生更准确的指导。

### 不是"我调了几个 API"——Praxis 知道"我们正在做 Phase 2 的 API 开发"

场景识别（横向）和任务感知（纵向）是两个正交维度。"门诊流程"属于 healthcare 场景——这是横向分类。"构建医院系统 Phase 2: API 开发"是当前任务——这是纵向时间轴。同一个场景可以出现在不同任务的不同阶段。

TaskContext 是一个 ~200 token 的极轻量任务感知层：task_id、task_name、current_phase、progress_summary、active_subtask、relevant_scenarios。session_end 时 LLM 自动推断进度变化——但置信度低于 0.7 的不自动更新（宁愿留给人来确认，也不做低置信度的自动修改）。

Praxis 还处理跨场景的同形异义词。"对接"在 API 开发场景中 = 系统集成，在干系人沟通场景中 = 会议确认需求，在数据处理场景中 = 格式转换。semantic-disambiguator 在 message_received 时用当前场景上下文消歧——减少因"听错意思"导致的学习噪声。

### 自己推动任务——不只是等待

Praxis 不只是被动响应 Hook 事件。驱动层赋予它主动推动任务的能力。

**TaskScheduler** 在每个 session_end 时评估：自主驱动是否启用？是否在静默时段？今日触发次数是否超限？然后决定：存在可并行子任务 → spawn 子 Agent；存在需要本 Agent 继续的任务 → 调度下个会话；存在需要等待外部反馈的任务 → 注册定期检查。

**SubagentManager** 管理并行子 Agent 的生命周期——构建上下文（子任务+陷阱预警+验收标准，不含父对话历史）、等待完成、超时检测、失败重试、汇总结果。最大并行 3，失败隔离：一个子 Agent 失败不影响其他子 Agent。

**HeartbeatMonitor** 每 5 分钟检查活跃任务：Level 1（nudge）→ 有活跃 session 时注入提醒；Level 2（wake）→ 无活跃 session 时主动请求唤醒；Level 3（escalate）→ 停顿超过 24 小时则标记 SUBTASK_BLOCKED 并推进外循环。

## 哲学的血统

Praxis 不是在一个周末里想出来的点子。它站在几个具体的、有名字的思想传统之上——这些不是"灵感来源"的模糊说法，而是可以直接在代码里找到对应物的设计决策。

**奎因（1948）："存在就是成为约束变元的值。"** 在奎因之前，哲学家争论"数字是否存在"、"属性是否存在"、"共相是否存在"——这些问题本身是混乱的。奎因的贡献是把"存在"问题从玄学变为逻辑：不要问 X 本身是否存在，去问你最好的理论在逻辑表述中**必须量化哪些实体**才能使理论成立。Praxis 的 `/praxis ontology` 命令就是这一原则的工程实现——它将 AI 的"信念"从隐式的权重分布变为可公开审计的结构化清单。"我们相信门诊流程（ps-clinic-flow）作为一个 sequence 存在，置信度 0.87，基于 23 次观察，当前为 crystallized 状态"——这就是 Praxis 的约束变元清单。

**Kroes & Meijers（2000s）：技术人工物的双重性质理论。** 一座桥既不是纯粹物理对象（一堆钢材和混凝土），也不是纯粹意向对象（"连接两岸"的抽象意图）。它是两者的不可还原的结合。Praxis 的 ProtoSequence 的 structure/function/teleological_mapping 三层结构是这一哲学立场的直接形式化。这个设计决策有实际操作后果——当用户改变了做法，Praxis 不比对步骤名称，而是检查功能面是否仍被满足。这使得 Praxis 能从"灵活性"和"学习"之间走出一条中间道路：接受等效实现，惩罚功能失败。

**中世纪的共相之争（12-14 世纪）：工程意义上的解决方案。** 实在论者说共相先于殊相存在（上帝心智中的模板），唯名论者说只有殊相真实而共相只是名称，概念论者说共相存在于事物之中并在心智中作为概念。Praxis 把这三派争论变成可配置的学习策略——`ontology_stance`：实在论（bootstrap 预设模板，适合冷启动）、唯名论（纯观察驱动，适合高风险领域——宁可慢不可错）、概念论（观察+LLM 归纳，适合常态运行）。运维者按领域风险配置混合权重。八百年的哲学争论，被一个 YAML 配置项操作化了。

**神经符号 AI（2010s-至今）：符号侧和神经侧的边界。** Praxis 的终极架构形态是神经符号的。符号侧——ProtoStructure、约束公理、统计验证器、确定性关系图传播规则——提供精确但脆弱的骨架，保证可靠性和可审计性。神经侧——LLM 的 transcript 分析、场景识别、语义理解、自然语言生成——提供灵活但不可靠的血肉，保证泛化能力和创造性。两侧不是"谁替代谁"，而是"谁负责什么"。7 源置信度融合中的 0.25/0.75 权重划分——LLM 信号合计 0.25，独立信号合计 0.75——就是这个边界的具体数值表达。

**波普尔（1934）：证伪主义，不是证实主义。** 科学理论的标志不是它被证实了多少次——任何理论都能找到支持证据。科学理论的标志是它承担了被证伪的风险。Praxis 的 ConceptVerifier 的对抗 prompt——"请为这个概念的反例辩护"——是这一原则的操作化。如果一个概念连一个有说服力的反例都找不到，它可能是对的，也可能是空洞的。如果一个概念能找到有力的反例——那它需要下调置信度。信念的价值不在于它被支持了多少次，而在于它经受住了多少次严肃的攻击。

## 5 分钟上手

```bash
.\scripts\install.ps1    # Windows — 自动检测 bun、注册 hooks、配置 AgentMemory
./scripts/install.sh      # macOS / Linux — 同上，额外注册 crontab
```

重启 Claude Code。Praxis 在后台工作。首次 session_start 初始化认知引擎，首次 session_end 提取第一批学习。之后每个 session 都是积累。
